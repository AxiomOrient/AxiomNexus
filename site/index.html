한 줄 정의

AxiomNexus는 AI가 바꾼 것을 실행하는 도구가 아니라, 그 변경을 왜 허용했는지 판단하고 기록하는 커널이다.

PHILOSOPHY

이 프로젝트가 지키는 단순함

적은 스킬, 더 선명한 역할

Nexus에는 로컬 스킬이 두 개뿐입니다. 나머지는 이미 검증된 범용 스킬을 그대로 씁니다. 로컬 복제본을 늘리지 않기 때문에 드리프트와 유지 비용이 줄어듭니다.

권위는 말이 아니라 기록에서 나온다

agent가 “끝났다”라고 말해도 그 자체로는 아무 의미가 없습니다. changed files, command results, artifacts 같은 관측 근거가 있어야 합니다.

Nexus는 어댑터만 로컬에 둔다

구현, 검증, 시나리오 테스트 같은 범용 동작은 upstream 스킬에 맡깁니다. Nexus는 그 결과를 계약과 evidence에 맞게 정리하는 얇은 층만 가집니다.

WHAT USERS GET

사용자가 이 스킬 시스템으로 얻는 것

거짓 완료를 줄이는 구조

완료 선언보다 근거 묶음을 먼저 보게 해서 과장된 완료를 줄입니다.

운영 경로와 진단 경로 분리

scheduler once와 run once <run_id>의 역할이 섞이지 않습니다.

업스트림 재사용

범용 skill 생태계를 그대로 쓰면서도 Nexus 계약을 잃지 않습니다.

레거시 없는 로컬 표면

복제된 skill 무더기 대신, 로컬 책임이 분명한 얇은 skill 두 개만 유지합니다.

SYSTEM FLOW

어떻게 움직이는가

01

범용 스킬이 실제 일을 한다

코드 변경, 최종 검증, 시나리오 테스트, plan 실행, plan-free loop는 upstream skill이 담당합니다.

02

Nexus Skill Adapter가 결과를 정리한다

pinned contract, lease, expected revision, unresolved obligations를 함께 보고, 결과를 evidence bundle로 바꿉니다.

03

Transition Executor가 intent를 만든다

evidence를 바탕으로 propose_progress, complete, block 중 무엇이 정직한지 결정하고 JSON만 반환합니다.

04

Kernel이 판정하고 Store가 기록한다

authority는 여기서 생깁니다. intent는 제안일 뿐이고, 최종 권위는 decision plus commit plus record에 있습니다.

LOCAL SKILLS

Nexus에 남겨둔 두 개의 로컬 스킬

transition-executor

authority gate

현재 work snapshot과 pinned contract revision을 읽고, 오직 유효한 TransitionIntent JSON만 반환합니다.

사용자가 얻는 것

  • 과장 없는 intent
  • schema에 맞는 JSON
  • complete/propose_progress/block의 엄격한 구분

핵심 기능

  • work, lease, expected rev 읽기
  • resolved obligations와 risks 정리
  • proof_hints를 intent로 압축

nexus-skill-adapter

thin local layer

어떤 범용 스킬을 써야 하는지 고르고, 그 결과를 Nexus가 믿을 수 있는 evidence bundle로 정리합니다.

사용자가 얻는 것

  • upstream 재사용과 local honesty를 동시에 확보
  • 입력은 최소, output은 명시적
  • 복제 skill 없이 Nexus 규칙 적용

핵심 기능

  • upstream skill 선택
  • contract, lease, rev, obligations 추가
  • changed_files, command_results, artifact_refs 정리

UPSTREAM SKILLS

Nexus가 직접 복제하지 않고 기대는 범용 스킬

build-write-code

한 번의 bounded code change와 verification evidence를 만듭니다.

check-final-verify

계약과 근거를 다시 대조해서 pass, blocked, inconclusive를 가릅니다.

test-run-user-scenarios

happy, failure, weird path를 실제 운영 시나리오처럼 태워 봅니다.

workflow-build-execute-plan

계획 문서가 있을 때 task row를 끝까지 먹으며 증거를 축적합니다.

control-build-until-done

계획 문서가 없을 때 done contract만 보고 구현과 검증을 반복합니다.

WHY THIS STAYS SMALL

더 많은 스킬이 아니라, 더 적은 책임 경계

Nexus는 구현 도구가 아닙니다. 그래서 구현용, 검증용, 시나리오용 skill을 다시 복제하지 않습니다. 이 프로젝트는 로컬 skill 표면을 최소화하고, authority가 필요한 지점만 남깁니다.

단순함은 비어 있는 상태가 아니라, 본질이 흐려지지 않는 상태입니다.
